## US-001: Динамическая активация региона

### Описание
Когда игрок входит в регион (например, "Тёмный лес"), система автоматически активирует агента региона без предварительного спавна.

### Предусловия
- Регион имеет тег `dark_forest` в конфиге.
- Агент региона имеет `spawn: condition` и условие `player_count >= 1`.

### Сценарий
```gherkin
Дано: Игрок "Player1" перемещается в регион "dark-forest-01"
Когда: Событие `player.entered_region` публикуется в EventBus
Тогда: 
  - Router находит blueprint `domain-dark-forest.md`
  - Проверяет условия спавна (совпадают)
  - Lifecycle создаёт контекст агента в Redis
  - Worker получает тик с событием
  - Агент загружает лор региона из Chroma
  - Агент оценивает окружение (поиск сущностей в радиусе)
```

### Критерии приёмки
- [ ] Время от события до активации агента < 200ms (p95)
- [ ] Контекст агента сохраняется в Redis с версионированием
- [ ] При повторном входе того же игрока контекст не пересоздаётся

## US-002: Адаптивный спавн и поведение NPC

### Описание
Агент региона, обнаружив игрока рядом с сущностью (волк), спавнит Task-агента для обработки встречи.

### Предусловия
- Агент региона активен (LOD >= 1).
- В радиусе 50м от игрока есть сущность типа `wolf`.

### Сценарий
```gherkin
Дано: Агент региона обрабатывает тик с позицией игрока
Когда: Правило `player_near_wolf` срабатывает
Тогда:
  - Агент вызывает `tool:spawn_task("wolf_encounter")`
  - Создаётся TaskAgent с контекстом: {player_id, wolf_id, region_id}
  - TaskAgent получает blueprint `task_wolf_encounter.md`
  - При событии `player.attack` TaskAgent запускает LLM-фазу 1 (механика)
  - После валидации JSON эмитит `combat.decided`
  - При `next_phase: narrative` запускается фаза 2 (описание)
  - Результат: `combat.rendered` → GameService показывает текст/эффекты
```

### Критерии приёмки
- [ ] Спавн Task-агента < 50ms
- [ ] LLM фаза 1 (механика) < 100ms (p95)
- [ ] Валидация JSON-ответа по схеме, автоматический retry при ошибке
- [ ] Фаза 2 (нарратив) выполняется асинхронно, не блокирует геймплей

## US-003: Контекстно-зависимая генерация нарратива

### Описание
Система генерирует описание событий на основе механического результата, лора региона и состояния окружения.

### Предусловия
- Фаза 1 (механика) завершилась успешно.
- В контексте есть ссылки на лор (`memory_ref`) и параметры окружения.

### Сценарий
```gherkin
Дано: Получен результат механики {hit: true, damage: 8, status: "killed"}
Когда: Запускается фаза 2 с шаблоном `combat_narrative.md`
Тогда:
  - Промт собирается из: механика + лор ("волки рассыпаются пеплом") + окружение ("лунный свет")
  - LLM возвращает строго валидный JSON: {description, effects[], mood}
  - GameService получает событие и отображает:
    * Текст: "Ваш меч рассекает призрачную плоть волка..."
    * Эффекты: ["particle:ash_burst", "sound:whisper"]
    * Настроение: "eerie"
```

### Критерии приёмки
- [ ] Генерация нарратива < 300ms (p95) или выполняется асинхронно
- [ ] Ответ строго соответствует схеме, иначе — fallback на шаблонный текст
- [ ] Поддержка мультиязычности через параметр `locale` в контексте

## US-004: Иерархический жизненный цикл агентов

### Описание
Агенты могут спавнить дочерние агенты для параллельной обработки подзадач. Родитель получает отчёт о завершении.

### Предусловия
- Родительский агент имеет в `tools` разрешение `spawn_agent`.
- Дочерний агент имеет `ttl` и `goal` в блупринте.

### Сценарий
```gherkin
Дано: Активен агент `task-combat-123`
Когда: Событие `skill.cast: fireball → tree`
Тогда:
  - Агент вызывает `spawn_agent("object-tree", context_snapshot)`
  - Создаётся ObjectAgent с контекстом: {object_id: "tree-42", damage: 15}
  - ObjectAgent обрабатывает:
    * Применяет урон к объекту
    * Проверяет `hp <= 0` → `emit_event("object.destroyed")`
    * Генерирует лут (если настроено)
  - При завершении эмитит `agent.child_resolved` родителю
  - Родитель обновляет свой контекст и решает следующие шаги
  - ObjectAgent удаляется из Redis по истечении `ttl`
```

### Критерии приёмки
- [ ] Передача контекста от родителя к ребёнку < 10ms
- [ ] Родитель получает `child_resolved` даже если ребёнок упал (fallback)
- [ ] Автоматическая очистка завершённых агентов (TTL + garbage collection)
- [ ] Максимальная глубина иерархии настраивается в конфиге (по умолчанию: 3)

## US-005: Динамическая адаптация уровня детализации (LOD)

### Описание
Уровень детализации агента меняется динамически в зависимости от нагрузки и значимости события.

### Предусловия
- В блупринте задан `lod_range: [min, max]`.
- Router имеет метрики плотности игроков и очереди событий.

### Сценарий
```gherkin
Дано: В регионе 1 игрок, очередь событий < 10/сек
Когда: Агент региона получает тик
Тогда:
  - Router рассчитывает LOD = 1 (правила + кэш, без LLM)
  - Агент использует детерминированные правила из `rule-engine`

Дано: В регионе 15 игроков, очередь > 100/сек
Когда: Агент региона получает тик
Тогда:
  - Router рассчитывает LOD = 3 (полный ReAct + LLM)
  - Агент загружает полный контекст, вызывает LLM для принятия решений
  - При превышении лимитов — автоматическое понижение LOD до 2
```

### Критерии приёмки
- [ ] Пересчёт LOD происходит не чаще 1 раза в 30 секунд на агента
- [ ] Понижение LOD не приводит к потере критических событий (приоритизация)
- [ ] Метрики `agent_lod_distribution` доступны в Prometheus

## Актёры и роли

| Актёр | Роль | Ответственность |
|-------|------|-----------------|
| **Player** | Пользователь | Генерирует события (`move`, `attack`, `interact`) |
| **DomainAgent** | Регион/Зона | Координация, спавн подзадач, агрегация метрик |
| **TaskAgent** | Конкретная задача | Обработка боя, квеста, диалога (короткий жизненный цикл) |
| **ObjectAgent** | Интерактивный объект | Реакция на воздействие (урон, использование) |
| **MonitorAgent** | Наблюдатель | Поиск аномалий, эскалация, сбор метрик |
| **AgentOrchestrator** | Оркестратор | Роутинг, пул воркеров, управление жизненным циклом |
| **LLMGateway** | Шлюз к моделям | Маршрутизация, кэширование, валидация, fallback |

## Глоссарий

| Термин | Определение |
|--------|-------------|
| **Blueprint** | MD-файл с метаданными и инструкциями для агента |
| **Context** | Состояние агента в моменте (Redis hot, PG cold) |
| **LOD** | Level of Detail: 0=спит, 1=правила, 2=кэш+простой LLM, 3=полный ReAct |
| **Phase 1 (Decision)** | Быстрый вызов LLM для механики (статы, броски, правила) |
| **Phase 2 (Narrative)** | Креативный вызов LLM для генерации описания и эффектов |
| **ToolRegistry** | Единый интерфейс для действий агентов (спавн, модификация, запрос) |
| **EventBus** | Шина событий (Kafka/Redis) для коммуникации между агентами и сервисами |
