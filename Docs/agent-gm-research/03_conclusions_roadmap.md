# Агентная архитектура GM: Выводы и дорожная карта

**Версия:** 1.0  
**Дата:** 2026-01-22  
**Статус:** Утверждено для планирования

---

## 1. Ключевые выводы исследования

### 1.1. Парадигмальный сдвиг

| Было (императивная) | Стало (декларативная) |
|---------------------|------------------------|
| Жёсткие `if-else` в коде игры | Логика в промптах + валидация по схеме |
| Предварительный спавн всех сущностей | Спавн по событию + blueprint (on-demand) |
| Монолитный игровой цикл | Иерархия агентов с параллельным выполнением |
| Один LLM-вызов на всё | Двухфазный конвейер: механика → нарратив |
| Конфиги отдельно от промптов | Единый источник: MD-блупринт |

**Результат**: Система становится гибче, масштабируемее и легче тестируемой. Изменение правил игры = правка MD-файла, не перекомпиляция.

### 1.2. Технические преимущества

| Преимущество | Реализация | Выгода |
|--------------|------------|---------|
| **Масштабируемость** | Stateless workers + шари по регионам | Линейный рост с нагрузкой, до 10к событий/сек |
| **Отказоустойчивость** | Fallback на rule-engine, retry, versioning | Работа при сбое LLM, восстановление контекста |
| **Гибкость правил** | Промпт-шаблоны + JSON Schema | Смена игровой системы без изменения кода |
| **Контроль затрат** | Кэш, LOD, routing моделей | 80% вызовов → кэш/детерминистика, экономия токенов |
| **Аудит и отладка** | Логирование каждого шага, корреляционные ID | Воспроизведение инцидентов, анализ поведения агентов |

### 1.3. Риски и стратегии смягчения

| Риск | Вероятность | Влияние | Стратегия |
|------|-------------|---------|-----------|
| **Высокая задержка LLM** | Средняя | Критическое | 2-фазный конвейер, кэш, fallback, мониторинг `llm_latency` |
| **Непредсказуемый ответ модели** | Высокая | Среднее | Строгая схема + retry + детерминированный fallback |
| **Взрывной спавн агентов** | Низкая | Критическое | `max_children`, `spawn_rate_limit`, алерты на `queue_depth` |
| **Сложность отладки промптов** | Высокая | Среднее | Версионирование блупринтов, A/B тестирование, логирование промптов (DEBUG) |
| **Стоимость LLM при росте** | Средняя | Высокое | LOD-адаптация, кэш, приоритизация, бюджет-алерты |

## 2. Архитектурные решения (утверждённые)

### 2.1. Что принимаем

✅ **Блупринты в MD**  
Единый формат для конфига + промпта + ограничений. Парсер: frontmatter (YAML) + H2-секции.

✅ **Двухфазный LLM-конвейер**  
Phase 1: механика (fast model, strict JSON). Phase 2: нарратив (creative model, async).

✅ **Stateless воркеры + Stateful контекст**  
Воркеры не хранят состояние. Контекст в Redis (hot) + PG (cold) с версионированием.

✅ **Иерархия агентов с наследованием контекста**  
Родитель → ребёнок: передача `region_id`, `player_id`, `memory_ref`, понижение LOD.

✅ **ToolRegistry с sandbox**  
Все действия через единый интерфейс: валидация, рейт-лимит, аудит.

✅ **Динамический LOD**  
Автоматическая адаптация детализации по плотности игроков и очереди событий.

### 2.2. Что откладываем

⏳ **Семантический кэш промптов**  
Точный кэш (hash(vars)) — обязательно. Семантический (векторный) — фаза 2.

⏳ **Автоматическая генерация блупринтов**  
Ручное создание/редактирование — сейчас. Генерация через LLM — после стабилизации.

⏳ **Кросс-регионная координация**  
Агенты работают внутри региона. Глобальные события — через GlobalAgent (фаза 1).

⏳ **Полная изоляция инструментов (WASM sandbox)**  
Сейчас: проверка прав + рейт-лимит. Полная изоляция — при работе с пользовательским кодом.

### Фаза 0: Подготовка (1 неделя)
| Задача | Результат | Ответственный |
|--------|-----------|---------------|
| Создать `shared/agent/` пакет | Базовые типы: `AgentContext`, `Blueprint`, `Tool` | Backend |

| Документировать стандарт промптов | `docs/prompts/README.md` | AI Engineer |

### Фаза 1: Ядро оркестратора (2-3 недели)
| Задача | Результат | Критерии приёмки |
|--------|-----------|------------------|
| `shared/agent/blueprint/` | Парсер MD, кэш в Redis, hot-reload | Загрузка 100 блупринтов < 1с, валидация схем |
| `shared/agent/context/` | StateManager (Redis+PG, versioning) | Load/Save < 5ms, optimistic locking работает |
| `services/agent-orchestrator/router.go` | Пространственный + типовой роутинг, LOD-фильтр | Соответствие маршрутизации тестам, метрики |
| Интеграция с `shared/eventbus` | `AgentEvent` wrapper, приоритеты | События доставляются, порядок сохраняется |

### Фаза 2: LLM-конвейер (2 недели)
| Задача | Результат | Критерии приёмки |
|--------|-----------|------------------|
| `shared/agent/llm/gateway.go` | Routing, кэш, валидация, fallback | Phase 1 < 100ms (p95), fallback при ошибке |
| Шаблоны промптов (Phase 1+2) | `configs/prompts/combat_*.md.json` | Валидация по схеме, воспроизводимость бросков |
| Интеграция с `rule-engine` | Фильтр до LLM | 70% запросов обрабатываются без LLM (на тестах) |
| Мониторинг | Метрики Prometheus, алерты | Дашборд `agent-gm` в Grafana |

### Фаза 3: Инструменты и жизненный цикл (2 недели)
| Задача | Результат | Критерии приёмки |
|--------|-----------|------------------|
| `shared/agent/tools/registry.go` | Registry, sandbox, built-in инструменты | Валидация прав, рейт-лимит, логирование |
| `internal/lifecycle.go` | Spawn/suspend/resume/terminate | Иерархия работает, `child_resolved` доставляется |
| Адаптивный LOD | Динамическое изменение по метрикам | Плавное переключение, нет потери событий |
| Нагрузочное тестирование | Отчёт: 1000 events/sec/shard | SLA выполнены, ресурсы в лимитах |

### Фаза 4: Миграция и пилот (2-3 недели)
| Задача | Результат | Критерии приёмки |
|--------|-----------|------------------|
| Конвертация `configs/gm_*.yaml` → MD | Блупринты для 3 регионов | Парсинг без ошибок, поведение идентично |

### Фаза 5: Продакшн и оптимизация (постоянно)

## 4. Рекомендации для архитектурного планирования

### 4.1. Приоритеты разработки
1. **Стабильность ядра**: `blueprint parser` + `context manager` + `eventbus integration` — основа всего.
2. **Безопасность LLM**: Валидация схем, fallback, sandbox — критично для продакшна.
3. **Наблюдаемость**: Метрики, логирование, трассировка — без этого невозможно отлаживать агентов.
4. **Гибкость конфигурации**: Вынос всех порогов, лимитов, моделей в конфиги — для оперативной настройки.

### 4.2. Командная структура
| Роль | Ответственность | Навыки |
|------|-----------------|---------|
| **Backend Engineer** | Ядро оркестратора, StateManager, EventBus | Go, Redis, PostgreSQL, Kafka |
| **AI Engineer** | Промпт-инжиниринг, LLM Gateway, валидация | Prompt design, JSON Schema, LLM API |
| **DevOps Engineer** | Деплой, мониторинг, автоскейлинг | Kubernetes, Prometheus, Grafana |
| **Game Designer** | Создание блупринтов, баланс правил, нарративы | GDD, нарративный дизайн, базовый YAML/MD |
| **QA Engineer** | E2E-тесты, нагрузочное тестирование, A/B | Go test, k6, Postman, анализ метрик |

### 4.3. Инфраструктурные требования
```yaml
# Минимальная конфигурация для пилота
orchestrator:
  instances: 3
  resources: {cpu: "500m", memory: "512Mi"}
  autoscaling: {min: 2, max: 10, target_cpu: 70%}

redis:
  cluster: true
  nodes: 3
  memory: "4Gi"  # для кэша контекстов

postgresql:
  version: 15
  storage: "100Gi"  # для архива контекстов
  backup: daily

llm:
  provider: ollama  # локально для пилота
  models: [qwen:7b, qwen:72b]
  gpu: {min: 1, recommended: 2}  # для 72B модели

monitoring:
  prometheus: true
  grafana_dashboards: [agent-gm, llm-performance]
  alertmanager: true
```

### 4.4. Критерии перехода к следующей фазе
- ✅ Все юнит-тесты проходят (покрытие > 80% для ядра)
- ✅ Нагрузочный тест: 1000 events/sec/shard, p95 latency < SLA
- ✅ E2E-сценарий "Тёмный лес" проходит без ошибок
- ✅ Документация обновлена, команда обучена
- ✅ Фича-флаг `agent_mode` протестирован (вкл/выкл без потери данных)

## Приложение: Чеклист для ревью архитектуры

- [ ] Блупринты хранятся в Git, парсятся при старте и горячо перезагружаются
- [ ] Контекст агента имеет версионирование, защита от гонок (optimistic locking)
- [ ] LLM-вызовы имеют таймаут, retry, fallback на детерминированную логику
- [ ] Все ответы LLM валидируются по строгой схеме перед использованием
- [ ] Инструменты имеют рейт-лимиты и проверку прав на уровне Registry
- [ ] LOD рассчитывается динамически, есть метрики распределения
- [ ] События между агентами идут через EventBus с приоритетами
- [ ] Логирование разделено: INFO для прода, DEBUG для разработки, корреляционные ID
- [ ] Конфигурация (таймауты, лимиты, модели) вынесена в YAML, не захардкожена
- [ ] Есть план отката: фича-флаг, canary deployment, мониторинг ключевых метрик
