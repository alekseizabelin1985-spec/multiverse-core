## 🎯 Краткое содержание

### Проблема
Текущая реализация Game Master основана на жёстких `if-else` и предварительном спавне сущностей. Это приводит к:
- Сложности масштабирования при росте игроков
- Трудностям изменения игровых правил (требуется перекомпиляция)
- Неэффективному использованию LLM (дорого, медленно)

### Решение
Переход к **агентной архитектуре** с ключевыми принципами:
1. **Блупринты в MD**: Единый источник для конфига + промпта + ограничений
2. **Event-driven спавн**: Агенты создаются по событию, не заранее
3. **Двухфазный LLM-конвейер**: 
   - Фаза 1 (механика): быстрый вызов, строгий JSON, fallback на rule-engine
   - Фаза 2 (нарратив): креативный вызов, асинхронно, не блокирует геймплей
4. **Иерархия агентов**: Global → Domain → Task → Object с наследованием контекста
5. **Динамический LOD**: Автоматическая адаптация детализации по нагрузке

### Ожидаемые результаты
| Метрика | Текущее | Целевое | Выгода |
|---------|---------|---------|---------|
| Масштабируемость | ~100 players/shard | ~1000 players/shard | ×10 рост нагрузки |
| Гибкость правил | Перекомпиляция | Правка MD-файла | Время изменения: дни → минуты |
| Стоимость LLM | 100% вызовов | ~20% вызовов (кэш+LOD) | Экономия ~80% токенов |
| Отказоустойчивость | Падение при сбое LLM | Fallback на rule-engine | Доступность 99.9% |
